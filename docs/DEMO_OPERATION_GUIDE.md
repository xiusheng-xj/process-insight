# Process Insight デモ操作手順書

お客様／社内向け操作デモ動画の撮影用ガイド。
本書のとおり起動・操作すれば、シーン1〜5を一貫して撮影できる。

- 対象データ: `PIDEMO-01`〜`PIDEMO-10`（`npm run seed:demo` で投入）
- 日付は撮影日（`CURRENT_DATE`）基準で自動計算されるため、**撮影直前に `seed:demo` を実行**すれば常に同じ見た目になる。
- 最終検証日: 2026-06-25（backend/frontend 起動・seed:demo・各シーン・Console Error 0 を確認済み）

---

## 0. 事前セットアップ

### 0-1. 起動

```bash
# ① バックエンド（ポート 6101）
cd backend
npm start            # → http://localhost:6101

# ② フロントエンド（ポート 6100）  ※別ターミナル
cd frontend
npm run dev          # → http://localhost:6100
```

### 0-2. デモデータ投入

```bash
cd backend
npm run seed:demo
```

> 前提: `schema.sql`〜`schema_v20.sql` と `seed.sql`（マスタ）が適用済みであること。
> `seed:demo` は `PIDEMO-%` のみを対象とし、冪等（再実行で削除→再投入）。既存データは壊さない。

### 0-3. 初回アクセス

ブラウザで `http://localhost:6100` を開く → **ユーザー名入力ダイアログ**が出るので任意の名前（例: 「デモ太郎」）を入力して「開始する」。
（編集ロック管理用の名前。ログインではない）

## デモ全体の流れ

```
案件一覧（シーン1）
   ↓ 全体像を見せる
複数案件ガント（シーン2）
   ↓ 月/週/日で俯瞰
予実比較＋アラート（シーン3：PIDEMO-03）
   ↓ 遅延の可視化
新規案件登録＋パターン適用（シーン4）
   ↓ 工程の自動生成・ロケーション/リソース入力
工程計画レビュー（シーン5：PIDEMO-06）
   ↓ Resource重複「要調整」→ 部門確認の喚起
```

所要時間の目安: 通しで 6〜8 分。

## シーン1：案件一覧

**画面**: ナビ「案件一覧」（`/projects`）

**操作**
1. 案件一覧を表示する。
2. `PIDEMO-01`〜`PIDEMO-10` の10件が並んでいることを確認。
3. ステータスバッジの色分けを見せる。

**見せどころ（状態の混在）**

| 案件 | 状態（effective / 健全性） | 見せ方 |
|------|----------------------------|--------|
| PIDEMO-01 | 作業中 / **計画通り** | 正常な進行 |
| PIDEMO-02 | 作業中 / **注意** | 軽微遅延 |
| PIDEMO-03 | 作業中 / **遅延** | 重大遅延（シーン3で深掘り） |
| PIDEMO-04 | **完了** / 計画通り | 完了案件 |
| PIDEMO-05 | **保留** / — | 保留案件 |
| PIDEMO-09 | 作業中 / **遅延** | 納期逼迫 |

**ナレーションメモ**
> 「Process Insight では、複数の製造案件を一覧で管理します。正常・注意・遅延・保留・完了といった状態が一目で分かります。」

## シーン2：複数案件ガント

**画面**: ナビ「プログラムガント」（`/gantt`）

**操作**
1. プログラムガントを開く。
2. 右上の **月 / 週 / 日** ボタンを順に切り替えて見せる。
3. 各案件のマイルストーンが時系列で並ぶ様子を見せる。

**見せどころ**
- 全案件を横断して工程の予定／実績が俯瞰できること。
- 「月」で全体、「週」「日」で詳細、とズームが切り替わること。
- 今日線（赤の縦線）と予定◆／実績●の関係。

**ナレーションメモ**
> 「複数案件を横断して、月・週・日の粒度で工程を俯瞰できます。設備や日程の重なりもここで気づけます。」

## シーン3：予実比較＋アラート（PIDEMO-03）

**画面**: 案件一覧 → **PIDEMO-03「産業用ロボット 組立セル」** をクリック → 案件詳細

**操作**
1. 案件詳細の工程テーブルを見せる。
2. 「差異」列で **`+N日（遅延）`**、実績未入力の超過行 **`未入力（+N日超過）`** を指す。
3. 画面上部の**アラートバナー**（赤系）を見せる。
4. 必要なら案件内ガントでも予定◆と実績●のズレを見せる。

**見せどころ**
- 予定に対する実績の遅延が数値と色で可視化されること。
- 重大遅延案件では `health=遅延`、critical アラートが出ること。
- 補助: **PIDEMO-09「半導体洗浄装置 量産対応」** は納期逼迫（要求納期が近い）の例として使える。

**ナレーションメモ**
> 「予定と実績の差異を自動計算し、遅延を色と数値で可視化します。一定以上の遅れはアラートとして通知されます。」

## シーン4：新規案件登録（マイルストーン自動生成）＋ロケーション/リソース

**画面**: 案件一覧（`/projects`）→「新規案件作成」

**操作（登録＝この時点でマイルストーン自動生成）**
1. 「新規案件作成」ボタンを押す。
2. 案件No（例: `PRJ-2026-DEMO`）、案件名、自部門担当者を入力。
3. **マイルストーンパターン**で「1：標準」または「2：簡易」「3：リピート」「4：EOL」を選択 → 「作成」。
4. **作成と同時に**選択したパターンのマイルストーンが自動生成され、そのまま案件詳細へ遷移してイベント一覧・ガントに表示される（別途「パターン適用」操作は不要）。
   - 標準=16件 / 簡易=13件 / リピート=6件 / EOL=6件。
   - ※ 既存案件にあとからパターンを当て直す場合は、従来どおり詳細の「パターン適用」ボタンが使えます。

**操作（ロケーション/リソース入力）**
5. 任意の工程の**編集**を開く（イベント編集モーダル）。
6. **ロケーション**プルダウン（埼玉工場／大阪工場／本社試験室／協力会社A／海外拠点）を選択。
7. **リソース/設備**プルダウン（MC-01／恒温槽／EMC試験室／組立ラインA／D部門 設計レビュー枠）を選択。
8. リソースを選ぶと、**ロケーション未設定なら既定ロケーションが自動補完**されることを見せる。
   - 選択中リソースの「既定ロケーション・capacity・部門」が下部に表示される。
9. 「更新」で保存。

**見せどころ**
- 案件作成時にパターンを選ぶだけで工程が一気に生成されること（追加操作不要）。
- 工程に「場所(location)」と「設備/能力枠(resource)」を別属性で持てること。
- リソース選択時のロケーション自動補完。

**ナレーションメモ**
> 「案件登録時にマイルストーンパターンを選ぶと、作成と同時にマイルストーンが自動生成されます。各工程には場所と設備を割り当てられ、設備を選ぶと既定の場所が自動で入ります。」

## シーン5：工程計画レビュー（Resource重複）（PIDEMO-06）

**画面**: 案件一覧 → **PIDEMO-06「標準コンベヤ リピート生産」** をクリック → 案件詳細 → 下部「工程計画レビュー」パネル

**操作**
1. PIDEMO-06 の詳細を開き、下部の **「工程計画レビュー」** パネルまでスクロール。
2. **Resource重複** が **⚠ 要調整** になっていることを見せる。
3. 明細「**KK日 が D部門 設計レビュー枠 に集中：3件 / capacity 2（+1 超過）**」と、対象 `PIDEMO-06 / 08 / 10` を指す。
4. **総合判定：要調整** と、確認喚起 **「💬 D部門と日程確認を行ってください。」** を見せる。
5. （対比）**PIDEMO-01** の詳細を開くと、工程計画レビューが **✔ 問題なし** になることを見せる。
6. （任意）プログラムガント（シーン2）で PIDEMO-06/08/10 の KK日が同一日に集中している様子と紐づける。

**見せどころ**
- 工程計画を「レビュー」する思想：問題を早期に発見する。
- 同一リソース・同一日程・同一工程が capacity を超えると「要調整」になること。
- システムは自動修正せず、**人間系の確認（D部門との日程調整）を促す**設計であること。
- 将来のレビュー項目（Location偏り／工程集中／担当者負荷 等）が淡色で見えており、拡張余地があること。

**ナレーションメモ**
> 「Process Insight は工程を管理するだけでなく、計画の妥当性をレビューします。設計レビュー枠のような限られたリソースに工程が集中すると、要調整として検出し、関係部門との確認を促します。判断は人に残し、気づきを与えるのが狙いです。」

## 撮影前チェックリスト

撮影開始前に上から順に確認する。

- [ ] **バックエンド起動**: `http://localhost:6101/health` が `{"status":"ok"}` を返す
- [ ] **フロントエンド起動**: `http://localhost:6100` が表示される
- [ ] **seed:demo 実行済み**: 直前に `npm run seed:demo` を実行した
- [ ] **案件一覧**: `PIDEMO-01`〜`PIDEMO-10` の10件が表示される
- [ ] **状態の混在**: 計画通り／注意／遅延／保留／完了 が一覧で見える
- [ ] **シーン3**: PIDEMO-03 で遅延（差異 `+N日`／未入力超過／赤アラート）が見える
- [ ] **シーン4**: 工程編集でロケーション・リソースのプルダウンが選べる
- [ ] **補完動作**: リソース選択でロケーションが自動補完される
- [ ] **シーン5（重複→要調整）**: PIDEMO-06 の工程計画レビューが「要調整」「D部門と日程確認を行ってください」になる
- [ ] **シーン5（対比）**: PIDEMO-01 の工程計画レビューが「問題なし」になる
- [ ] **Console Error**: ブラウザ DevTools（F12）→ Console にエラーが出ていない
- [ ] **ブラウザ準備**: 画面拡大率・ウィンドウ幅を整え、不要なタブ／拡張機能の通知をオフにした

## 想定トラブルと対処

| 症状 | 原因 | 対処 |
|------|------|------|
| ユーザー名ダイアログが出る | 初回アクセス（ログインではない） | 任意の名前を入力して「開始する」 |
| **白画面／`504 (Outdated Optimize Dep)`** がConsoleに多発 | vite dev サーバーの依存キャッシュが古い | フロントを Ctrl+C → `rm -rf frontend/node_modules/.vite` → `npm run dev` で**再起動**してから撮影 |
| `Port 6100/6101 is already in use` | 旧プロセスが残存 | `node scripts/kill-port.js 6100`（または 6101）で解放してから再起動 |
| 工程計画レビューが全部「問題なし」になる | `seed:demo` 未実行、または REVIEW-D 割当前のデータ | `cd backend && npm run seed:demo` を再実行 |
| 案件一覧に PIDEMO が出ない | `seed:demo` 未実行 / DB 未接続 | `seed:demo` 実行・`backend/.env` の DB 接続設定を確認 |
| 画面で `/api` 通信エラー | バックエンド未起動 | `cd backend && npm start`（6101）を起動 |
| パターン適用で工程が0件になる | `seed.sql` 未適用（パターン3/4の工程定義なし） | `seed.sql` を適用。通常は `seed:demo` が3/4を補完するため発生しない |
| 日付が想定とずれて見える | 日付は `CURRENT_DATE` 基準で自動計算 | 撮影直前に `seed:demo` を再実行（その日基準で再生成される） |
| レビューを編集後に更新したい | 工程編集後は自動再評価されるが、手動でも可 | レビューパネル右上の「再評価」ボタンを押す |
